### Project Initialized — June 2026

**Approach:** Monorepo with Next.js 15 (storefront-ui) and FastAPI (analytics-brain).
Two Alibaba Cloud Function Compute deployments. WebSocket pipeline for all live data.
No polling anywhere in the system.

**Architecture decisions made before first line of code:**
- FastAPI over Next.js API routes — native WebSocket support, Pydantic validation,
  cleaner separation from frontend
- Qwen3.5-plus for multimodal logo analysis (replaces separate qwen-vl call)
- qwen3-max for decision cycles, qwen-plus for product descriptions
- Redis (Tair) as the fast state layer — all telemetry, snapshots, pending actions
- JSON patch (jsonpatch) for delta execution — no full state rewrites
- BrandGuardRules generated by Qwen at brand creation time — Qwen authors
  its own defense rules, interceptor fires Qwen's own words as warnings

**Key insight documented:**
The Subconscious Interceptor has three layers — brand guard (Qwen-authored),
business constraints (merchant-set), system safety (hard blocks).
Each layer has different severity and different source of authority.

**Stack confirmed:**
- storefront-ui: Next.js 15, TypeScript, Tailwind, Framer Motion, Zustand, Zod
- analytics-brain: FastAPI, Python 3.10, Pydantic v2, async Redis, httpx
- Infrastructure: Alibaba Cloud FC, Redis (Tair), OSS
- AI: Qwen Cloud (qwen3.5-plus, qwen3-max, qwen-plus)
- Deploy: Serverless Devs (s.yaml)

**Next:** Sprint 1 — merchant auth → logo upload → brand generation → live store

### Repo Restructure + Dead Code Cleanup — June 10, 2026

**Approach:** The monorepo had drifted from its intended layout during an earlier
move — root-level docs and configs (README, LICENSE, docker-compose.yml,
.gitignore, slash commands) had ended up inside `storefront-ui/`, the frontend
Zod schema file was nested one directory too deep
(`storefront-ui/storefront-ui/types/schemas.ts`), and a junk `{app/{api/{qwen`
empty directory tree existed from a brace-expansion `mkdir` that PowerShell
doesn't support. Restored the documented layout: docs/configs at root,
`storefront-ui/` is the Next.js app only, `analytics-brain/` is the FastAPI
backend only.

Deleted frontend code that predated the FastAPI decision — Next.js API routes
(`app/api/`), JSON-patch logic (`lib/patches/`), interceptor middleware
(`lib/middleware/`), Redis clients (`lib/redis/`), and direct Qwen calls
(`lib/qwen/`). All of that logic belongs to the backend. Removed the matching
dependencies (`fast-json-patch`, `ioredis`, `qrcode`) from package.json and
added the actual frontend stack (`zod`, `framer-motion`). Deleted the stale
`types/index.ts` — `types/schemas.ts` is the single Zod source of truth
mirroring the backend's Pydantic models.

**Qwen calls:** none — structural task only.

**Problems:** The mangled nesting meant the correct schemas.ts and a stale
type file coexisted in different places; git had the old tree deleted but the
new tree fully untracked, so nothing was recoverable from history.

**Solutions:** Verified every file's content before moving or deleting
(rather than trusting filenames), confirmed remaining app files had no imports
into deleted directories, and verified .gitignore rules with `git check-ignore`
before staging — no env files or private files can reach the public repo.

**Edge cases tested:** grep for imports of deleted modules in surviving code
(none), gitignore coverage for `**/.env` and `.venv/` at both root and
backend levels (all excluded).

**Next:** Sprint 1 build order step 1 — merchant auth (JWT, bcrypt,
httpOnly cookie) in analytics-brain.

### Merchant Auth — June 10, 2026

**Approach:** Email + password auth with bcrypt hashes and a JWT carried in an
httpOnly cookie (`elevate_session`) — never localStorage, so XSS can't lift the
token. Four routes: `POST /auth/signup`, `POST /auth/login`, `POST /auth/logout`,
`GET /auth/me`. Onboarding state lives directly on the merchant row
(`onboarding_status`) — no separate session model. Signup collects store info,
so the status advances straight to `logo_upload`.

Slug generation follows the spec: "Emma Fashion" → `emma-fashion`, and only on
a DB collision does it retry with a random suffix (`emma-fashion-f66d`). Guard
rails: symbol-only or non-ASCII store names fall back to `store` + suffix
rather than producing an empty slug.

Supporting fixes along the way:
- `config.py` had `database_url` declared *after* the return statement in
  `get_settings()` — unreachable code, the DB engine would have crashed on
  first use. Moved into Settings properly, added `jwt_secret` (required —
  startup fails loudly if missing) and token expiry settings.
- `MerchantDB` gained `slug` (unique, indexed), `category`, `description`,
  `is_live` to match the Pydantic `Merchant` schema.
- Two stale scaffold routers (`onboarding`, `api`) import schema classes that
  no longer exist — they're excluded from `main.py` with explicit notes until
  their rewrite steps, so the app boots clean instead of dying on import.
- Dev convenience: `create_all` on startup in development only; production
  uses Alembic against RDS. A DB outage at startup logs loudly but doesn't
  prevent the API from serving.
- Zod mirror updated in the same task: `MerchantSchema.logo_url` accepts the
  pre-upload empty string; added `MerchantCreateSchema` / `MerchantLoginSchema`
  for frontend form validation.

**Qwen calls:** none — auth is deterministic infrastructure.

**Problems:**
1. The venv had no pip (uv-managed) and zero packages installed.
2. `greenlet==3.5.1` (SQLAlchemy async dependency) ships a broken wheel for
   this Windows/py3.10 combo — `DLL load failed importing _greenlet`.
3. The integration test deadlocked silently: an in-memory SQLite engine's
   pooled connection stayed bound to the `asyncio.run()` event loop while
   FastAPI's TestClient ran its own loop.

**Solutions:**
1. Installed everything via `uv pip install -r requirements.txt`.
2. Pinned `greenlet==3.1.1` in requirements.txt with a do-not-unpin comment.
3. Switched the test DB to a file-backed temp SQLite with `NullPool` — every
   connection is fresh, nothing crosses event loops.

**Edge cases tested (10-check integration suite, full HTTP layer):**
duplicate email → 409; slug collision → suffixed; wrong password vs unknown
email → byte-identical 401s (no account enumeration); logout kills session;
missing cookie → 401; tampered JWT → 401; expired JWT → 401; short password
and malformed email → 422; bcrypt handles malformed stored hashes without a
500; response bodies never contain password material.

**Next:** STS token endpoint for direct OSS upload (build order step 2),
then qwen-vl-max logo analysis. Postgres smoke test of auth once Docker
Desktop is running.
